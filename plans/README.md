   Son los tres de menor riesgo: tipado, seguridad CI, y empaquetado de vectores.

   `--strict` desde su creación.

## Findings considered and rejected

### From deep audit (001–007 cycle)

- **pip-audit reported 11 vulnerabilities**: all in the local dev/CI
  virtualenv, none in the library's runtime dependencies (the core has zero).
  Not a package finding — no plan.
- **`_sanitizar_mensaje` bare-numeric bypass** (`exceptions.py:11`): the PII
  regex requires a `-[0-9kK]` suffix, so an over-length all-digits value is
  logged unsanitized. Marginal — only triggers on malformed, over-length,
  non-RUT numeric input. Note-and-reject.

### From direction audit (008–012 cycle)

- **MkDocs documentation site deploy**: already done — `.github/workflows/docs.yml`
  deploys to GitHub Pages on push to master. `site/` and `docs/api/` exist.
  Removed from v1.8 pending items.
- **Benchmark regression CI gate**: already done —
  `.github/workflows/benchmark-regression.yml` with 20% threshold
  (non-blocking). Removed from v1.8 pending items.
- **Python 3.14 in CI**: already in matrix with `continue-on-error: true`.
  Removed from v1.8 pending items.
- **pytest-benchmark conftest incompatibility**: `tests/benchmarks/conftest.py`
  references `pytest_benchmark_update_machine_info` hook not present in the
  installed pytest-benchmark version. Only affects local runs when benchmarks
  directory is included; CI uses `--benchmark-only` which bypasses it.
  Workaround documented: use `--ignore=tests/benchmarks` locally. No plan
  needed unless the hook is removed upstream.

### From direction audit (013–018 cycle)

- **Django Ninja / Strawberry GraphQL integrations**: mentioned as potential
  expansion but rejected — the existing 4 integrations (pydantic, fastapi,
  pandas, polars) cover the dominant Chilean Python ecosystem. Adding more
  without user demand is premature.
- **SRI (Servicio de Impuestos Internos) integration**: on the long-term
  roadmap (`ROADMAP.md:95-96`) but rejected for this cycle — requires an
  external HTTP dependency and API contract with a government service. Too
  high-risk for a solo-maintainer library without an official SRI API.
- **Full i18n (multi-language error messages)**: infrastructure exists (plan
  012) but completing translations for all error messages into 5+ languages
  is deferred — English is done, more languages when there's demand.
- **CLI `--watch` mode / interactive TUI**: on the long-term roadmap but
  rejected for this cycle — the CLI is feature-complete for the current use
  cases. A TUI adds significant complexity (curses/textual) for unclear gain.
